## Summary

Phase 11 is a pure SQL migration phase. There is no application code to write beyond the `.sql` migration files and the Supabase CLI setup commands. The deliverable is two migration files applied to the existing Supabase project via `supabase db push`.

The project already has one migration (`0001_create_bookings.sql`) with a known style: plain `CREATE TABLE` with no supabase-specific syntax and no RLS. Phase 11 extends this pattern by adding RLS to the new tables. All seed values are confirmed from reading `lib/pricing.ts` and `lib/extras.ts` directly.

The critical design decision already resolved in CONTEXT.md is whether `pricing_globals` goes in `0002` or a separate file. Research recommends bundling it into `0002` (one logical unit: all pricing config tables) rather than a confusingly-named `0002b`. Migration `0003` stays as `coverage_zones` only.

**Primary recommendation:** Write `0002_create_pricing_config.sql` covering both `pricing_config` and `pricing_globals` tables (schema + RLS + seed), and `0003_create_coverage_zones.sql` (schema + RLS, no seed). Apply with `supabase db push` after CLI setup.

## Architecture Patterns

### Migration File Structure

```
supabase/
└── migrations/
    ├── 0001_create_bookings.sql       # existing
    ├── 0002_create_pricing_config.sql # Phase 11: pricing_config + pricing_globals
    └── 0003_create_coverage_zones.sql # Phase 11: coverage_zones
```

No `supabase/config.toml` currently exists (only `migrations/` directory is present). Running `supabase init` in the project root will generate `supabase/config.toml` and `supabase/.gitignore`. These should be committed so CLI commands work correctly in future CI/CD.

### Pattern 1: RLS-enabled table with public-read

Consistent with Supabase best practices for tables that the client (booking wizard) reads without authentication:

```sql
-- Source: Supabase RLS documentation pattern
CREATE TABLE table_name (
  ...
);

ALTER TABLE table_name ENABLE ROW LEVEL SECURITY;

CREATE POLICY "public_read" ON table_name
  FOR SELECT USING (true);
```

The `bookings` table (migration 0001) has NO RLS. The new tables in Phase 11 must add RLS explicitly because the booking wizard will query `pricing_config` and `coverage_zones` via the anon key (public client).

### Pattern 2: Singleton table (pricing_globals)

`pricing_globals` holds exactly one row. Enforced at application layer (Phase 14 API routes will always UPDATE, never INSERT). A `CHECK` constraint is optional but recommended to prevent a second row being inserted accidentally:

```sql
-- Option A: No DB-level enforcement (simplest)
CREATE TABLE pricing_globals (...);

-- Option B: Unique sentinel column (prevents accidental second row)
CREATE TABLE pricing_globals (
  id INTEGER PRIMARY KEY DEFAULT 1,
  CHECK (id = 1),
  ...
);
```

Recommendation: Use Option B (`id INTEGER PRIMARY KEY DEFAULT 1, CHECK (id = 1)`) — it costs nothing and prevents an entire class of silent bugs where the API reads row 0 instead of the updated row.

### Pattern 3: JSONB for GeoJSON polygons

`coverage_zones.geojson` stores full GeoJSON Feature or Geometry objects. JSONB is the correct type (binary, indexed, queryable). No special index needed in Phase 11 — Phase 14 will do point-in-polygon checks in application code using `@turf/boolean-point-in-polygon` (already in `package.json`), not via PostGIS.

### Anti-Patterns to Avoid

- **Do not add write RLS policies in Phase 11.** Write access is intentionally deferred to API route middleware (Phase 14). Adding `authenticated` write policies now creates confusion about where the authorization boundary lives.
- **Do not use `SERIAL` or `BIGSERIAL` for pricing_globals id.** Use `INTEGER DEFAULT 1` with `CHECK (id = 1)` to make the singleton semantics explicit.
- **Do not create a separate migration file per table in the same logical unit.** `pricing_config` and `pricing_globals` are one logical feature — bundle them.

## Common Pitfalls

### Pitfall 1: Supabase CLI not initialized before `db push`

**What goes wrong:** Running `supabase db push` without first running `supabase init` and `supabase link --project-ref` fails with no project linked error.

**Why it happens:** `supabase db push` requires `supabase/config.toml` which `init` generates, and requires the project to be linked.

**How to avoid:** The wave must explicitly include steps: `supabase init` (if config.toml absent), `supabase login`, `supabase link --project-ref <ref>`, then `supabase db push`.

**Warning signs:** `Error: Cannot find project config. Run supabase init` or similar.

### Pitfall 2: Migration 0001 was applied via SQL Editor, not CLI

**What goes wrong:** `supabase db push` will attempt to apply `0001_create_bookings.sql` again if the CLI's migration history table (`supabase_migrations.schema_migrations`) is empty, causing a "table already exists" error.

**Why it happens:** The existing `bookings` table was created manually via Dashboard SQL Editor, not via `supabase db push`. Supabase CLI migration history is separate from what tables exist.

**How to avoid:** After linking, run `supabase migration repair --status applied 0001_create_bookings` to tell the CLI that `0001` is already applied, before running `db push` for migrations 0002 and 0003.

**Warning signs:** `ERROR: relation "bookings" already exists` during `supabase db push`.

### Pitfall 3: NUMERIC precision and EUR decimal values

**What goes wrong:** Seed values like `2.80` inserted as floating-point TEXT may lose precision if column type is `FLOAT` instead of `NUMERIC`.

**Why it happens:** PostgreSQL `FLOAT8` (used in `bookings` for lat/lng) is IEEE 754 and imprecise for financial values.

**How to avoid:** Use `NUMERIC` or `NUMERIC(10,2)` for all monetary/rate columns. SQL literal `2.80` is exact in NUMERIC.

### Pitfall 4: RLS blocks anon reads if policy not created

**What goes wrong:** After `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`, all rows are invisible to the anon key until a `SELECT` policy is added. Missing the `CREATE POLICY` statement means Phase 12 reads zero rows and silently fails.

**Why it happens:** RLS default is deny-all when enabled.

**How to avoid:** Always pair `ENABLE ROW LEVEL SECURITY` with `CREATE POLICY "public_read" ... FOR SELECT USING (true)` in the same migration file.

## Code Examples

Verified patterns from existing project files and Supabase documentation conventions:

### Existing migration style (0001_create_bookings.sql)

```sql
-- Plain CREATE TABLE, no supabase-specific syntax
-- Filenames: NNNN_snake_case_description.sql
CREATE TABLE bookings (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  created_at timestamptz DEFAULT now() NOT NULL,
  ...
);
-- No RLS in 0001 — new tables add RLS
```

### pricing_config table + RLS + seed (migration 0002, part 1)

```sql
CREATE TABLE pricing_config (
  vehicle_class  TEXT PRIMARY KEY,
  rate_per_km    NUMERIC(10,2) NOT NULL,
  hourly_rate    NUMERIC(10,2) NOT NULL,
  daily_rate     NUMERIC(10,2) NOT NULL
);

ALTER TABLE pricing_config ENABLE ROW LEVEL SECURITY;

CREATE POLICY "public_read" ON pricing_config
  FOR SELECT USING (true);

INSERT INTO pricing_config (vehicle_class, rate_per_km, hourly_rate, daily_rate) VALUES
  ('business',     2.80, 55, 320),
  ('first_class',  4.20, 85, 480),
  ('business_van', 3.50, 70, 400);
```

### pricing_globals table + RLS + seed (migration 0002, part 2)

```sql
CREATE TABLE pricing_globals (
  id                  INTEGER PRIMARY KEY DEFAULT 1 CHECK (id = 1),
  airport_fee         NUMERIC(10,2) NOT NULL DEFAULT 0,
  night_coefficient   NUMERIC(5,4)  NOT NULL DEFAULT 1.0,
  holiday_coefficient NUMERIC(5,4)  NOT NULL DEFAULT 1.0,
  extra_child_seat    NUMERIC(10,2) NOT NULL,
  extra_meet_greet    NUMERIC(10,2) NOT NULL,
  extra_luggage       NUMERIC(10,2) NOT NULL
);

ALTER TABLE pricing_globals ENABLE ROW LEVEL SECURITY;

CREATE POLICY "public_read" ON pricing_globals
  FOR SELECT USING (true);

INSERT INTO pricing_globals (
  id, airport_fee, night_coefficient, holiday_coefficient,
  extra_child_seat, extra_meet_greet, extra_luggage
) VALUES (
  1, 0, 1.0, 1.0, 15, 25, 20
);
```

### coverage_zones table + RLS (migration 0003, no seed)

```sql
CREATE TABLE coverage_zones (
  id         UUID      DEFAULT gen_random_uuid() PRIMARY KEY,
  name       TEXT      NOT NULL,
  geojson    JSONB     NOT NULL,
  active     BOOLEAN   NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT now() NOT NULL
);

ALTER TABLE coverage_zones ENABLE ROW LEVEL SECURITY;

CREATE POLICY "public_read" ON coverage_zones
  FOR SELECT USING (true);
```

## Open Questions

1. **Exact Supabase project-ref for `supabase link`**
   - What we know: The project was already set up (Phase 10 confirmed Supabase client works with the project)
   - What's unclear: The project-ref string — found in Supabase Dashboard URL: `https://supabase.com/dashboard/project/<project-ref>`
   - Recommendation: Implementer must retrieve this from Dashboard before running `supabase link`. It is not stored in any repo file.

2. **Whether 0001 has a timestamp prefix in CLI migration format**
   - What we know: `0001_create_bookings.sql` uses sequential numbering, not ISO timestamp format that CLI prefers
   - What's unclear: Whether `supabase migration repair` references the filename stem or requires timestamp format
   - Recommendation: Run `supabase migration list` after linking to see what format the CLI expects, then use that in `migration repair`. If CLI doesn't recognize `0001_create_bookings`, the repair command may need `--version 0001` or similar flag.

3. **Whether `pricing_globals` singleton constraint should use `id = 1` check or trigger**
   - What we know: `CHECK (id = 1)` is simple and standard PostgreSQL
   - What's unclear: Whether Phase 16 admin UI (editing globals) will need to handle INSERT vs UPDATE logic
   - Recommendation: Use `id = 1 CHECK` — it is the simplest approach. Phase 14 API will use `UPDATE ... WHERE id = 1` which is unambiguous.
