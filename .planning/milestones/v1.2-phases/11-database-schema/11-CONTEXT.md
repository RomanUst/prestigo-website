<decisions>
## Implementation Decisions

### pricing_config schema scope
- Full schema now — includes ALL fields Phase 16 will need to edit:
  - Per-vehicle-class rates (`vehicle_class` TEXT PRIMARY KEY, `rate_per_km` NUMERIC, `hourly_rate` NUMERIC, `daily_rate` NUMERIC)
  - 3 rows: `business`, `first_class`, `business_van`
- Global pricing fields go in a separate **`pricing_globals`** table (single row):
  - `airport_fee` NUMERIC — flat surcharge for airport pickup/dropoff
  - `night_coefficient` NUMERIC — multiplier applied at calculation time
  - `holiday_coefficient` NUMERIC — multiplier applied at calculation time
  - `extra_child_seat` NUMERIC — EUR surcharge per booking
  - `extra_meet_greet` NUMERIC — EUR surcharge per booking
  - `extra_luggage` NUMERIC — EUR surcharge per booking
- Avoids ALTER TABLE in future phases

### Seed values
- **pricing_config** — seed from current placeholder values in `lib/pricing.ts`:
  - business: rate_per_km=2.80, hourly_rate=55, daily_rate=320
  - first_class: rate_per_km=4.20, hourly_rate=85, daily_rate=480
  - business_van: rate_per_km=3.50, hourly_rate=70, daily_rate=400
- **pricing_globals** — seed from `lib/extras.ts` placeholders + zero defaults for new fields:
  - extra_child_seat=15, extra_meet_greet=25, extra_luggage=20
  - airport_fee=0, night_coefficient=1.0, holiday_coefficient=1.0
- Operator will update real production values via admin dashboard once Phase 16 is live

### Migration application method
- Use **Supabase CLI** (`supabase db push`) — proper migration tracking going forward
- Phase 11 must include full CLI setup: install CLI, `supabase login`, `supabase link --project-ref`, then `supabase db push`
- Migration files created in `supabase/migrations/` for version control (consistent with existing `0001_create_bookings.sql`)
- Migration numbering: `0002_create_pricing_config.sql`, `0003_create_coverage_zones.sql`
  - Note: if a `pricing_globals` table is added as part of migration 0002, it goes in the same file or a dedicated `0002b_` / bundled into `0002`

### coverage_zones schema
- Full schema for the zones table (will be needed by Phase 14 API and Phase 16 UI):
  - `id` UUID DEFAULT gen_random_uuid() PRIMARY KEY
  - `name` TEXT NOT NULL
  - `geojson` JSONB NOT NULL — GeoJSON polygon
  - `active` BOOLEAN NOT NULL DEFAULT true
  - `created_at` TIMESTAMPTZ DEFAULT now() NOT NULL
- No seed data — empty table, operator draws zones via UI in Phase 16

### RLS policy
- Both tables: RLS enabled, public-read policy (consistent with bookings table pattern)
- Write access: authenticated admin only (enforced at API route layer in Phase 14, not via additional RLS rules in Phase 11)

### Claude's Discretion
- Whether to bundle `pricing_globals` into migration `0002` or use a separate `0002b` file
- Exact column type precision (NUMERIC vs NUMERIC(10,2))
- Supabase CLI install method (npm global vs brew vs direct binary)
- Whether to commit `supabase/.gitignore` and `supabase/config.toml` generated by CLI init

</decisions>
