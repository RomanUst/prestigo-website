The user must set environment variables in the Vercel Dashboard. Claude cannot access Vercel Dashboard.

**Step-by-step instructions for the user:**

1. Open **Vercel Dashboard** > select the prestigo-website project
2. Go to **Settings** > **Environment Variables**
3. **First, check which vars already exist.** The Google Maps keys may already be set from v1.0 deployment. If they exist, verify the values are correct and scope is Production only. Do not create duplicates.

4. For each of the following 7 variables, click **Add New**:
   - Set **Environment** to **Production** ONLY (uncheck Preview and Development)
   - Enter the variable name and value exactly as shown

| Variable Name | Value Source | Notes |
|---------------|-------------|-------|
| `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` | Google Cloud Console > APIs & Services > Credentials > your client key | May already exist — update if needed, ensure Production scope only |
| `GOOGLE_MAPS_API_KEY` | Google Cloud Console > APIs & Services > Credentials > your server key | API restriction only, NO HTTP referrer |
| `SUPABASE_URL` | Supabase Dashboard > Project Settings > API > Project URL | Format: `https://xxxx.supabase.co` |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase Dashboard > Project Settings > API > service_role key | Starts with `eyJ...` — never expose to browser |
| `STRIPE_SECRET_KEY` | Stripe Dashboard > Developers > API keys > Secret key (live mode) | Starts with `sk_live_` |
| `RESEND_API_KEY` | Resend Dashboard > API Keys | Starts with `re_` |
| `MANAGER_EMAIL` | Your email address for booking alerts | Plain email address, e.g. `roman@rideprestige.com` |

5. Do NOT set `STRIPE_WEBHOOK_SECRET` in this phase — it is set in Phase 8 after registering the live webhook endpoint.

6. After adding all 7 vars, scroll through the list and verify:
   - All 7 variable names are present
   - Each is scoped to **Production** only (not Preview or Development)
   - No duplicate entries for any variable

**Critical check:** `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` has `NEXT_PUBLIC_` prefix (client-side). `GOOGLE_MAPS_API_KEY` does NOT have the prefix (server-side only). Do not mix these up.

  <read_first>prestigo/.env.example</read_first>
  <action>
Two actions for the user: (A) submit Resend DNS records, (B) verify Vercel deployment.

**Part A — Resend DNS records:**

1. Open **Resend Dashboard** (resend.com/domains)
2. If `rideprestige.com` is not already added:
   - Click **Add Domain**
   - Enter `rideprestige.com`
   - Click **Add**
3. Resend will display DNS records (typically 2-3 records: SPF TXT record, DKIM CNAME records)
4. Go to your **domain registrar** (where rideprestige.com DNS is managed)
5. Add each DNS record exactly as shown in the Resend Dashboard:
   - For TXT records: copy the exact value
   - For CNAME records: copy the name and value exactly
   - Do NOT append the root domain to subdomain names if your registrar adds it automatically
6. Back in Resend Dashboard, click **Verify** (it may take 24-48 hours to propagate — that is expected and verified in Phase 9)

**Part B — Verify Vercel deployment:**

The git push from Plan 01 should have triggered a Vercel redeployment automatically. If not, trigger one manually:

1. Open **Vercel Dashboard** > Deployments
2. Confirm the latest deployment is **green** (successful build)
3. If no recent deployment exists, click **Redeploy** on the latest production deployment

Then verify the site is live:

```bash
curl -s -o /dev/null -w "%{http_code}" https://rideprestige.com
```

Expected output: `200`

If the deployment failed, check the build logs in Vercel Dashboard > Deployments > click the failed deployment > Build Logs. Common issues:
- Missing env var referenced at build time — check that `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` is set (it's the only build-time var)
- TypeScript errors — should not happen since no code changed, but check logs
