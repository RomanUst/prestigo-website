Create `supabase/migrations/023_v14_schema_foundation.sql` with the following 6 steps, in order. The complete SQL is provided below — write it verbatim, not paraphrased.

**STEP 1 — Add `leg` column to bookings:**
```sql
ALTER TABLE bookings
  ADD COLUMN IF NOT EXISTS leg TEXT NOT NULL DEFAULT 'outbound'
    CHECK (leg IN ('outbound', 'return'));
```
Default `'outbound'` auto-backfills all existing one-way rows. No separate UPDATE needed.

**STEP 2 — Replace single-column UNIQUE with composite UNIQUE:**
```sql
ALTER TABLE bookings DROP CONSTRAINT IF EXISTS bookings_payment_intent_id_key;
ALTER TABLE bookings
  ADD CONSTRAINT bookings_payment_intent_id_leg_key
    UNIQUE (payment_intent_id, leg);
```
The old constraint name `bookings_payment_intent_id_key` is the Postgres auto-generated name from `payment_intent_id text UNIQUE` in `0001_create_bookings.sql`. Use `DROP CONSTRAINT IF EXISTS` as safety net.

**STEP 3 — Add self-referential `linked_booking_id` FK:**
```sql
ALTER TABLE bookings
  ADD COLUMN IF NOT EXISTS linked_booking_id UUID REFERENCES bookings(id) ON DELETE SET NULL;
```
Must be SET NULL (not CASCADE) — cancelling one leg must not delete the other.

**STEP 4 — Add per-leg amount columns to bookings:**
```sql
ALTER TABLE bookings
  ADD COLUMN IF NOT EXISTS outbound_amount_czk INTEGER,
  ADD COLUMN IF NOT EXISTS return_amount_czk   INTEGER;
```
Both nullable. One-way bookings leave these NULL; round-trip bookings populate both.

**STEP 5 — Add `return_discount_pct` to pricing_globals:**
```sql
ALTER TABLE pricing_globals
  ADD COLUMN IF NOT EXISTS return_discount_pct NUMERIC(5,2) NOT NULL DEFAULT 10;
```
Default 10 means 10% discount on return leg. The singleton `pricing_globals` row (id=1) gets this value automatically.

**STEP 6 — Create atomic round-trip booking RPC:**
Create the `create_round_trip_bookings(p_outbound JSONB, p_return JSONB)` PL/pgSQL function that:
1. Inserts outbound row with `leg = 'outbound'`, captures `v_outbound_id`
2. Inserts return row with `leg = 'return'` and `linked_booking_id = v_outbound_id`, captures `v_return_id`
3. Updates outbound row: `SET linked_booking_id = v_return_id WHERE id = v_outbound_id`
4. Returns `TABLE(outbound_id UUID, return_id UUID)`

Use `CREATE OR REPLACE FUNCTION` so re-running is idempotent. The function body uses `VALUES (...)` syntax (not `SELECT ... FROM`). The complete column list for each INSERT must match the bookings table columns used by the webhook: `booking_reference, payment_intent_id, leg, booking_type, trip_type, origin_address, origin_lat, origin_lng, destination_address, destination_lat, destination_lng, passengers, luggage, pickup_date, pickup_time, vehicle_class, distance_km, amount_czk, outbound_amount_czk, return_amount_czk, extra_child_seat, extra_meet_greet, extra_luggage, client_first_name, client_last_name, client_email, client_phone, flight_number, terminal, special_requests, status, booking_source`.

The return row INSERT additionally includes `linked_booking_id` in its column list.

Each JSONB value is extracted with `p_outbound->>'column_name'` and cast as needed: `::float8` for lat/lng/distance, `::integer` for passengers/luggage/amounts, `::boolean` for extras. Text columns need no cast. Use `COALESCE(p_outbound->>'status', 'confirmed')` and `COALESCE(p_outbound->>'booking_source', 'online')` for defaults.

The exact SQL for this function is in `.planning/phases/23-database-schema-foundation/23-RESEARCH.md` under "Code Examples > Complete Migration File" starting at STEP 6. Copy it verbatim.
