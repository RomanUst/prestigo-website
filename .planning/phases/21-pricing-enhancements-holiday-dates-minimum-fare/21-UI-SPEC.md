## Typography

All admin UI uses Montserrat for body/UI; Cormorant Garamond is reserved for display/headings. Exactly 2 weights: 300 (body, labels) and 500 (CTAs/buttons). Visual distinction between body and labels is achieved via letter-spacing, not weight.

| Role | Font | Size | Weight | Line Height | Letter-spacing | Colour |
|------|------|------|--------|-------------|----------------|--------|
| Body / input value | Montserrat | 13px | 300 | 1.8 | 0.5px | `var(--offwhite)` / `#F5F2EE` |
| UI label / field label | Montserrat | 11px | 300 | 1.4 | 0.3em (≈3.3px) | `var(--warmgrey)` / `#9A958F` |
| Section sub-heading | Montserrat | 11px | 300 | 1.4 | 0.4em | `var(--warmgrey)` / `#9A958F` |
| Button / CTA text | Montserrat | 11px | 500 | n/a | 3px | `var(--offwhite)` or `var(--anthracite)` |

Rules:
- All labels and field headings: `text-transform: uppercase`
- Never use bold (700+) weights — lightness signals luxury (STYLEGUIDE.md rule)
- New section heading for holiday dates ("HOLIDAY DATES"): 11px Montserrat 300 uppercase, letter-spacing 0.4em, `var(--warmgrey)` — identical to `headerLabelStyle` in PricingForm.tsx
- Date pills in the holiday list: 13px Montserrat 300, `var(--offwhite)` — same as body/input value style

Source: STYLEGUIDE.md section 3 Typography; PricingForm.tsx `headerLabelStyle` and `inputBaseStyle` constants.

## Focal Point

The admin pricing form has two visual anchors:

1. **Primary anchor — "SAVE PRICING" button:** The copper-bordered ghost CTA at the bottom of the form is the dominant action target. It is the only element that transitions to a copper fill on hover, drawing the operator's eye downward to confirm and submit all changes.

2. **Secondary anchor — "+ ADD DATE" button in Section D:** The copper-bordered ghost button in the new holiday dates section is the primary call-to-action within Section D. It shares the same ghost-to-fill hover pattern as "SAVE PRICING" but is scoped to the date management interaction, making it the focal point of Section D specifically.

No other interactive elements use the copper border treatment. All other inputs use `var(--anthracite-light)` borders at rest and `var(--copper)` only on focus — ensuring visual hierarchy is preserved.

## Component Inventory

### Modified Components

#### PricingForm.tsx — additions

Two additions to the existing form, each rendered as a new card section below the existing Section B (Global Parameters):

---

##### Section C — "MIN FARE PER CLASS" (new column in Section A, not a separate card)

The minimum fare is added as a new column in the existing Section A vehicle class rates grid.

**Grid change:** Section A grid expands from `'160px 1fr 1fr 1fr'` to `'160px 1fr 1fr 1fr 1fr'` — adding a fourth data column.

**New column header:**
```
Label: "MIN FARE"
Sub-label: "EUR min"
Style: identical to existing column headers (headerLabelStyle)
```

**New input per vehicle class row:**
```
type="number"
step="0.01"
min="0"
placeholder="0"
style: identical to existing inputBaseStyle
width: 100px
```

- Registered with `registerNumeric(config.${index}.min_fare, config-${index}-min_fare)` — same focus border pattern as existing numeric inputs
- A value of `0` means "no minimum fare floor" — label "(no floor)" in warm-grey 10px shown beneath the input when value is `0`

State: Managed by react-hook-form alongside existing config fields. Included in PUT payload as `min_fare` per vehicle class row.

---

##### Section D — "HOLIDAY DATES" (new card below Section B)

Card style: identical to existing `cardStyle` (`background: var(--anthracite-mid); border: 1px solid var(--anthracite-light); border-radius: 4px; padding: 24px`).

**Card header:**
```
Label: "HOLIDAY DATES"
Style: headerLabelStyle (11px Montserrat 300, uppercase, letter-spacing 0.4em, var(--warmgrey))
Sub-text: "EUR / km"  →  no sub-text needed; instead show: "Trips on these dates apply the holiday coefficient."
Sub-text style: 11px Montserrat 300, var(--warmgrey), marginTop: 4px
```

**Holiday date list:**

Each entry in the list renders as a row:
```
Layout: flex row, align-items: center, gap: 8px
- Date string (13px Montserrat 300, var(--offwhite)): e.g. "2026-12-25"
- Remove button: icon-only, lucide-react <X> size 14px, color var(--warmgrey)
  - On hover: color changes to #f87171
  - aria-label="Remove {date}"
  - No border, no background, cursor: pointer
  - min touch target: 32px × 32px (padding: 8px)
```

Row gap between entries: 8px.

If the holiday list is empty:
- Show: "No holiday dates configured." — 11px Montserrat 300, `var(--warmgrey)`, italic style optional
- (No separate empty state card — renders inline within Section D)

**Add date row** (rendered below the list, always visible):
```
Layout: flex row, align-items: center, gap: 8px
- <input type="date">
  style:
    background: var(--anthracite)
    border: 1px solid var(--anthracite-light)
    border-radius: 2px
    color: var(--offwhite)
    font-family: var(--font-montserrat)
    font-size: 13px
    padding: 8px 12px
    width: 160px
  on focus: border-color: var(--copper)
  on blur: border-color: var(--anthracite-light)
- "+ ADD DATE" button
  style:
    border: 1px solid var(--copper)
    color: var(--copper)
    background: transparent
    font-family: var(--font-montserrat)
    font-size: 11px
    font-weight: 500
    text-transform: uppercase
    letter-spacing: 0.3em
    padding: 8px 16px
    border-radius: 0
    cursor: pointer
  on hover: background: var(--copper), color: var(--anthracite)
  disabled state: when input is empty or date is already in the list
```

**Interaction rules:**
- Clicking "+ ADD DATE" with a valid, non-duplicate date: appends date string (ISO `YYYY-MM-DD`) to the local `useState<string[]>` array; clears the date input
- Clicking "+ ADD DATE" with an empty input: no-op (button is disabled)
- Clicking "+ ADD DATE" with a duplicate date: no-op (button is disabled; date input border turns `#f87171` to signal duplicate)
- Clicking × on a date row: removes that entry from the array immediately (no confirmation — remove is in-memory until Save Pricing)
- The holiday dates array is included in the form PUT payload under `globals.holiday_dates`
- State is managed with `useState<string[]>` outside react-hook-form (dynamic list) — included in PUT submission alongside `handleSubmit(onSubmit)` data

**Validation at submit time:** Dates must match `/^\d{4}-\d{2}-\d{2}$/`. Client enforces this via `<input type="date">` browser validation. Server validates via Zod `z.array(z.string().regex(...))`.

## Copywriting Contract

All copy follows PRESTIGO brand voice: declarative, no hype, no hedging, sentence brevity, no exclamation marks, UPPERCASE labels with wide letter-spacing.

| Element | Copy |
|---------|------|
| Primary CTA | "SAVE PRICING" (unchanged; submits all pricing data including new fields) |
| Section C column header | "MIN FARE" |
| Section C column sub-label | "EUR min" |
| Min fare hint when value is 0 | "(no floor)" |
| Section D card heading | "HOLIDAY DATES" |
| Section D card sub-text | "Trips on these dates apply the holiday coefficient." |
| Add date button label | "+ ADD DATE" |
| Remove date aria-label | "Remove {date}" (e.g. "Remove 2026-12-25") |
| Section D empty state | "No holiday dates configured." |
| Save success feedback | "Pricing saved." |
| Save error feedback | "Save failed. Try again." |
| Duplicate date validation | (no text — input border turns red; button stays disabled; no additional label needed) |

No destructive confirmation needed for holiday date removal — the removal is in-memory until "SAVE PRICING" is clicked. Operator can discard by navigating away.

No new empty state for the pricing page — the page always has vehicle class rows (seeded at deployment). The only empty state is Section D's empty holiday list (handled inline above).
