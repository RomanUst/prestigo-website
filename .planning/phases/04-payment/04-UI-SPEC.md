## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|

| sm | 8px | Label-to-input gap, error message gap |

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#1C1C1E` `var(--anthracite)` | Page background, wizard shell background |
| Secondary (30%) | `#2A2A2D` `var(--anthracite-mid)` | Booking summary card, Payment Element container, confirmation card |
| Tertiary surface | `#3A3A3F` `var(--anthracite-light)` | Borders, dividers, input borders (non-error), section separators |
| Accent (10%) | `#B87333` `var(--copper)` | See reserved-for list below |
| Accent hover | `#D4924A` `var(--copper-light)` | `.btn-primary` hover fill only |
| Accent pale | `#E8B87A` `var(--copper-pale)` | Italic display text, shimmer gradient midpoint |
| Primary text | `#F5F2EE` `var(--offwhite)` | Headings, price figures, input values, button text |
| Secondary text | `#9A958F` `var(--warmgrey)` | Labels after rendering, body copy, secondary info |
| Destructive | `#C0392B` | Inline validation error text and input border only |

Accent (`var(--copper)`) reserved for:
1. Label text (ALL uppercase labels throughout the form)
2. `.btn-primary` border (Pay button border, Back button is `.btn-ghost` with `var(--anthracite-light)` border)
3. `.btn-primary` hover background fill
4. `focus-visible` outline ring (2px, 4px offset — already global in `globals.css`)
5. Stripe Appearance API — `labels` and `focusOutline` color
6. Booking reference badge accent border or copper-line decoration on confirmation page
7. Copper-line divider element (`.copper-line` class) used as section decorators

Accent is NEVER used as: background fill, card background, or general border color.

## Layout Contract

### Step 6 — Payment (within BookingWizard)

- Container: `max-w-[560px] mx-auto` — centered narrow column, checkout modal feel
- PriceSummary panel: hidden entirely at Step 6 (no right-column, no mobile bar)
- BookingWizard mobile bottom bar: hidden at Step 6 (Pay button is inline in the form)
- Stacking order (top to bottom, all in single column):
  1. Booking summary block (collapsible or static read-only card)
  2. Horizontal divider (`1px solid var(--anthracite-light)`, 32px vertical margin)
  3. Payment Element (Stripe) — full width within column
  4. Payment error message zone (inline, below Payment Element)
  5. Pay button (`.btn-primary`, full width on mobile, auto width on desktop)
- Booking summary block content order:
  1. "YOUR JOURNEY" label (`.label` class, copper)
  2. Route: origin → destination (or `N hours` for Hourly) — 13px warmgrey
  3. Date + time — 13px warmgrey
  4. Vehicle class + passenger count — 13px warmgrey
  5. Extras list (each extra on its own line with `+CZK X` suffix) — shown only if extras selected
  6. Divider
  7. Total in CZK (20px offwhite, weight 400) + EUR equivalent below it (13px warmgrey)
- Currency format: `CZK 2,450` primary (large), `(€98)` secondary (small, below or inline)
- Pay button label includes amount: `PAY CZK 2,450` (uppercase, letter-spaced per `.btn-primary`)

### Confirmation Page — `/book/confirmation`

- Full-page standalone route, NOT inside BookingWizard shell — no wizard chrome
- Page background: `var(--anthracite)` matching site background
- Centered content column: `max-w-[640px] mx-auto px-4`
- Stacking order (top to bottom):
  1. Copper-line accent (`.copper-line`) — decorative top separator
  2. PRESTIGO wordmark (`.wordmark`) — links back to homepage
  3. Small vertical gap (32px)
  4. Confirmation status label (`.label`): "BOOKING CONFIRMED" or "QUOTE REQUEST SENT"
  5. Booking reference — display size (Cormorant Garamond 26–36px, weight 300, offwhite), e.g. `PRG-20260330-0042`
  6. Booking summary card (`var(--anthracite-mid)` background, `border-radius: 4px`, padding 24px)
  7. Divider
  8. Action buttons row: "Print / Save PDF" + "Add to Calendar" + "Contact Us" (all `.btn-ghost`)
  9. Bottom copper-line accent
- Quote variant (`?type=quote`): same layout, reference uses `QR-` prefix, status label reads "QUOTE REQUEST SENT"
- After landing: Zustand store reset (no booking data visible on refresh)

Source: `04-CONTEXT.md` Step 6 Layout + Confirmation Page sections

## Interaction States

### Pay Button

| State | Visual |
|-------|--------|
| Default | `.btn-primary` — transparent bg, copper border, offwhite text |
| Hover | Copper background fill, anthracite text (matches `.btn-primary:hover`) |
| Disabled (while processing) | `opacity: 0.4`, `cursor: not-allowed` — matches existing VehicleCard disabled pattern |
| Disabled (pre-validation) | Same disabled appearance — active before card details are complete |
| Re-enabled on error | Returns to default state — user can retry |

### Payment Error

- Rendered below the Payment Element, above the Pay button
- Copy format: "Payment unsuccessful. [Stripe error message]. Please check your details and try again."
- Color: `#C0392B` for error text — matches existing inline validation error pattern from Step5Passenger
- No modal, no toast — inline only, persistent until user corrects and retries

### Payment Processing Overlay

- Use Stripe Payment Element's built-in loading state (spinner within element)
- No custom overlay — matches Claude's Discretion grant in `04-CONTEXT.md`
- Pay button disabled immediately on click (STEP6-04 requirement)

### Step Transition

- Step 6 entry: `animate-step-enter` class (stepFadeUp 0.3s) — same as all wizard steps
- Source: `globals.css` `.animate-step-enter` keyframe, `04-CONTEXT.md` reusable patterns

## Destructive Actions

No destructive actions in Phase 4. The only irreversible action is payment itself, which is not destructive — it is the primary success path. Error recovery (re-try) is handled inline without any confirmation dialog.

Quote submission (`/api/submit-quote`) is similarly non-destructive and requires no confirmation.

Store reset on confirmation page arrival is automatic (no user-facing action).

## Accessibility Requirements

- Pay button: `aria-disabled="true"` when processing (matches existing stepper `aria-disabled` pattern from Phase 1 decision log)
- Stripe Payment Element: built-in ARIA support — no custom attributes needed
- Booking reference on confirmation: `aria-label="Booking reference [value]"` for screen readers
- Action buttons on confirmation: explicit `aria-label` for print, calendar, contact
- Focus order on Step 6: summary block (read-only, skip with Tab) → Payment Element → Pay button → Back button
- `focus-visible` outline already global: `2px solid var(--copper)`, `outline-offset: 4px`
