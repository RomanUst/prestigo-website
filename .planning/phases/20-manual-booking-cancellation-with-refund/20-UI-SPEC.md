## Spacing Scale

Declared values — project uses a custom named scale, NOT 8-point multiples. These are the project tokens from STYLEGUIDE.md:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, dot indicators, tight inline padding |
| sm | 8px | Badge padding, compact element spacing (e.g. `padding: 4px 8px` on StatusBadge) |
| md | 24px | Label-to-content, card padding, form field gap |
| lg | 48px | Section padding |
| xl | 80px | Page horizontal margins |

For Phase 20 specifically:

| Context | Value | Rule |
|---------|-------|------|
| Modal overlay padding | 24px | Matches existing card padding (`--space-md`) |
| Form field row gap | 16px | Between form field rows inside ManualBookingForm modal |
| Form section gap | 24px | Between logical field groups (trip details / passenger details) |
| Modal max-width | 640px | Centered card; fits all required fields without horizontal scroll |
| Button height (modal CTAs) | 44px | Minimum touch target per UX-01 (Phase 22 enforces globally; Phase 20 adopts proactively for modal buttons) |
| Expanded row cancel button | 44px min-height | Same touch target rule |

Exceptions: 6px (`--space-xs` project variant) is acceptable for tight icon-to-label gaps inside buttons.

Source: STYLEGUIDE.md section 6 Spacing. Phase 22 UX-01 touch target requirement adopted proactively for modal buttons.

## Typography

All admin UI uses Montserrat for body/UI elements; Cormorant Garamond is reserved for display/headings.

| Role | Font | Size | Weight | Line Height | Letter-spacing | Colour |
|------|------|------|--------|-------------|----------------|--------|
| Body / form input value | Montserrat | 13px | 300 | 1.8 | 0.5px | `var(--offwhite)` / `#F5F2EE` |
| UI label / field label | Montserrat | 11px | 400 | 1.4 | 0.3em (≈3.3px) | `var(--warmgrey)` / `#9A958F` |
| Modal heading | Cormorant Garamond | 26px | 300 | 1.2 | 0 | `var(--offwhite)` / `#F5F2EE` |
| Button / CTA text | Montserrat | 11px | 500 | n/a | 3px | `var(--offwhite)` or `var(--anthracite)` |

Rules:
- All labels and field headings: `text-transform: uppercase`
- Never use bold (700+) weights — lightness signals luxury (STYLEGUIDE.md rule)
- Modal subheadings (e.g. "TRIP DETAILS", "PASSENGER DETAILS"): 11px Montserrat 400, uppercase, `var(--copper)` / `#B87333`, letter-spacing 0.3em — matches the copper section-label pattern

Source: STYLEGUIDE.md section 3 Typography. Existing `DetailField` component in BookingsTable confirms 11px labels at `letterSpacing: '0.3em'` with `var(--warmgrey)` and 13px values with `var(--offwhite)`.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#1C1C1E` (`var(--anthracite)`) | Page backgrounds, modal card background, input backgrounds |
| Secondary (30%) | `#2A2A2D` (`var(--anthracite-mid)`) | Elevated card surface, modal overlay card |
| Accent (10%) | `#B87333` (`var(--copper)`) | Accent reserved for: primary CTA button background, modal section-label text, copper gradient decorative line on modal header, "MANUAL" source badge |
| Destructive | `#f87171` (red, same tint as `cancelled` StatusBadge) on `#2a1a1a` bg | Cancel confirmation button, warning text in cancellation modal |

Accent reserved for specific elements only (never used as a fill color for large surfaces):
1. "Add Booking" primary button background
2. "Confirm Cancel + Refund" CTA — copper border ghost variant (NOT filled, to signal caution not confirmation)
3. Modal section-label text color (e.g. "TRIP DETAILS")
4. Source badge for `manual` bookings in the bookings table row

Borders: `1px solid var(--anthracite-light)` / `#3A3A3F` for all input fields and modal card border — matches existing admin input pattern.

Modal overlay: `position: fixed; inset: 0; background: rgba(0,0,0,0.6); z-index: 50` — matches RESEARCH.md Pattern 4.

Source: STYLEGUIDE.md section 2 Colour Palette. StatusBadge.tsx colour values confirm destructive tint. RESEARCH.md Pattern 4 confirms overlay rgba value.

## Component Inventory

### New Components

#### 1. ManualBookingForm (modal)

**Trigger:** "New Booking" button in the bookings page header (top-right, ghost variant).

**Layout:** `position: fixed; inset: 0` overlay with centered card (`max-width: 640px; max-height: 90vh; overflow-y: auto`). Card background `var(--anthracite-mid)`. Copper gradient accent line at top of card (2px height).

**Form structure — two labeled sections:**

Section A — "TRIP DETAILS" (copper section label):
- Trip type: `<select>` — options: Transfer / Hourly / Daily
- Pickup date: `<input type="date">`
- Pickup time: `<input type="time">`
- Origin address: `<input type="text">`
- Destination address: `<input type="text">` + label "(optional for hourly/daily)" in warm-grey 10px
- Vehicle class: `<select>` — options: Business / First Class / Business Van
- Passengers: `<input type="number" min="1" max="20">`
- Luggage: `<input type="number" min="0" max="20">`
- Price (CZK): `<input type="number">` — label "PRICE (CZK) — OPERATOR ENTERED"
- Optional fields (collapsible or always shown): Hours, Return date, Flight number, Terminal, Special requests

Section B — "PASSENGER DETAILS" (copper section label):
- First name, Last name: side-by-side on one row (2-column grid)
- Email, Phone: side-by-side on one row (2-column grid)

**Input style** (exact, from existing BookingsTable date filters):
```
background: var(--anthracite)
border: 1px solid var(--anthracite-light)
border-radius: 2px
color: var(--offwhite)
font-family: var(--font-montserrat)
font-size: 13px
font-weight: 300
padding: 8px 12px
```

**Modal footer buttons (right-aligned):**
- "Cancel" — ghost button (`border: 1px solid var(--anthracite-light); color: var(--warmgrey)`)
- "Create Booking" — primary CTA (`background: var(--copper); color: var(--offwhite)`)
- Both buttons: `min-height: 44px; padding: 0 24px`

**Loading state:** "Create Booking" button shows "Creating..." text and `disabled` state during submission. No spinner needed.

**Error state:** Inline error text below the form footer — 11px Montserrat 300, `#f87171`. Shows server error message verbatim.

**Success:** On 201 response, close modal and call `fetchBookings()` to refresh table.

#### 2. CancellationModal (inline in BookingsTable)

**Trigger:** "Cancel Booking" button in the expanded booking row. State: `pendingCancel: Booking | null`.

**Variant A — Stripe-paid booking** (`payment_intent_id !== null`):

Heading (26px Cormorant Garamond 300): "Cancel Booking"

Body text (13px Montserrat 300, `var(--warmgrey)`):
"This booking was paid online. Cancelling will issue a full refund to the client's card. This action cannot be undone."

Warning line (11px Montserrat 400 uppercase, `#f87171`): "A FULL STRIPE REFUND WILL BE ISSUED."

Buttons (right-aligned):
- "Keep Booking" — ghost, `border: 1px solid var(--anthracite-light); color: var(--warmgrey)`, min-height 44px
- "Confirm Cancel + Refund" — ghost copper variant, `border: 1px solid var(--copper); color: var(--copper)`, min-height 44px

**Variant B — Manual booking** (`payment_intent_id === null`):

Heading: "Cancel Booking"

Body text: "This booking was created manually and has no payment record. Cancelling will mark the booking as cancelled."

Buttons (right-aligned):
- "Keep Booking" — ghost grey
- "Cancel Booking" — ghost copper variant (`border: 1px solid var(--copper); color: var(--copper)`)

Note: copper ghost (not filled red) is used for both cancel CTAs to avoid false urgency signals inconsistent with the luxury brand voice. The warning text covers the destructive severity for Stripe-paid cases.

**Modal layout:** Same overlay and card pattern as ManualBookingForm. `max-width: 480px` (confirmation dialogs are narrower than forms).

**Loading state:** Active button shows "Cancelling..." text and `disabled`.

**Error state:** Same inline footer error pattern as ManualBookingForm.

## Copywriting Contract

All copy follows brand voice: declarative, no hype, no hedging, sentence brevity, no exclamation marks.

| Element | Copy |
|---------|------|
| Primary CTA — create booking | "Create Booking" |
| Primary CTA — bookings page header | "New Booking" |
| Cancel CTA — Stripe-paid confirmation | "Confirm Cancel + Refund" |
| Cancel CTA — manual booking confirmation | "Cancel Booking" |
| Dismiss modal / keep booking | "Keep Booking" |
| ManualBookingForm modal heading | "New Booking" |
| CancellationModal heading (both variants) | "Cancel Booking" |
| Refund warning body (Stripe-paid) | "This booking was paid online. Cancelling will issue a full refund to the client's card. This action cannot be undone." |
| Refund warning label | "A FULL STRIPE REFUND WILL BE ISSUED." |
| Manual cancel body | "This booking was created manually and has no payment record. Cancelling will mark the booking as cancelled." |
| ManualBookingForm — submitting | "Creating..." |
| CancellationModal — cancelling | "Cancelling..." |
| Error state — generic | "Something went wrong. Please try again." |
| Error state — Stripe refund failed | Server message verbatim (e.g. "Stripe refund failed: charge_already_refunded") |
| Error state — booking not found | "Booking not found." |
| Error state — terminal state cancel | "This booking cannot be cancelled." |
| Manual source badge | "MANUAL" |
| Price field label | "PRICE (CZK) — OPERATOR ENTERED" |
| Destination address helper | "(optional for hourly/daily)" |

Empty state: no new empty state needed — the BookingsTable already has one from Phase 16. ManualBookingForm opens empty by design (operator fills it).

No toast infrastructure. Errors appear inline. Per STATE.md decision for v1.3: "alert() for status update failures in admin — single-operator v1.3, no toast infrastructure needed." Phase 20 extends this: inline modal footer errors (not `alert()`) are preferred for modal-scoped errors since the modal is already open.
