Plan 02 changes only the CLIENT presentation; server-side behaviour is enforced by Phase 49/50 logic that already keys on `booking_source`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-51-04 | Tampering | Operator forces refund modal on GNet row by editing client state | accept | Even if the client UI were tricked into rendering Variant A for a GNet booking, the cancel API receives only `{ id, leg? }` — no refund flag. The server reads `booking_source` from Supabase (source of truth) and skips Stripe for GNet rows (existing behaviour from Phase 49/50 — verified by `grep "booking_source === 'gnet'" prestigo/app/api/admin/bookings/`). UI-only suppression is defence-in-depth, not the primary control. |
| T-51-05 | Repudiation | Operator claims they did not cancel a GNet booking | accept | The cancel action writes a status change to `bookings` (existing audit trail) and triggers a GNet status push via `lib/gnet-client.ts` (Phase 50). Both events are logged. No new audit gap is introduced. |
| T-51-06 | Denial of Service | Malicious admin spams cancel actions on GNet rows | accept | `getAdminUser()` guard on `/api/admin/bookings/cancel` (existing). No rate-limit regression: Plan 02 changes only UI strings and conditional rendering. |
| T-51-07 | Information Disclosure | GNet-specific copy reveals partner relationship | accept | Copy is visible only to authenticated admins. The fact that Prestigo accepts GNet bookings is not a secret (it's published partner info). |
</threat_model>
